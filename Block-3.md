# STL

## Общие вопросы

### 1. Из чего состоит STL?

- контейнеры
- алгоритмы
- итераторы
- функторы
- адаптеры

### 2. Перечислите контейнеры STL, которые вы знаете/пользуетесь.

- vector
- unordered_map
- unordered_set
- deque
- array

### 3. Как устроен std::vector?

Динамический массив в непрерывной области памяти
Хранит внутри указатель на данные, size, capacity
при переполнение выделяется новый буффер данных в 2 раза больше, перекладывает все данные туда, старый буффер освобождается

Вставка в конец O(1) амортизировано
Удаления с конца O(1)
Вставка в середину O(n)
Итераторы не стабильные если случилась релокация при добавлении
При вставках/удалениях в середину все итераторы начиная с позиции невалидны

### 4. Как устроен std::forward_list

Односвязный список
Каждый узел хранит указатель только на следующий элемент списка
В самой структуре только указатель на первый элемент

Вставка после конкретного элемента O(1)
Поиск O(n)
size O(n)

### 5. Как устроен std::map 

Ассоциативный контейнер, построенный на красно-черном дереве
Хранит пары (ключ, значение)
Упорядочено по ключу

Все базовые операции O(log n)
Итераторы стабильные

### 6. Как устроен std::unordered_map?

Хэш-таблица с разрешением коллизий методом цепочек
Массив бакетов, каждый бакет - связный список элементов с одинаковым хэшем
Время вставки O(1) в хорошем случае O(n) в худшем
При переполнении бакетов - делается rehash, все итераторы не валидны

### 7. Как использовать свои стуктуры данных в качестве ключей в ассоциативных контейнерах?

Для построенных на дереве структур - требуется строгое слабое упорядочение, то есть нужно мочь построить компаратор для любых двух обьектов, или перегрузить оператор </>

Для построенных на хеш таблицах - реализации хеш функции и operator ==

### 8. Чем отличается std::map от std::unordered_map? Что быстрее?

std::map - на дереве, ключи упорядочены, операции O(log n) гарантированно
std::unordered_map - на хеш-таблице, порядка нет, операции O(1) в среднем, в худшем O(n)

Строго говоря unordered_map быстрее, но зависит от качества хеш функции, размера данных и взаимодействии с данными 

### 9. Что такое хеш-функция? Чем отличается хорошая хеш-функция от плохой?

Функция, которая принимает данные произвольного размера и возвращает значение фиксированного, оно и называется хешем

Хорошая функция:
- детерминированная (одинаковый вход -> одинаковый результат)
- равномерная (разные входы распределеются равномерно по диапазону, без кластеров)
- быстрая (O(длина входа))

### 10. Что такое коллизия? Как разрешается коллизция в STL в std::unordered_map? Какие методы разрешения коллизий ты еще знаешь? Приведи пример коллизии.

Коллизия - два разных ключа дали одинаковый бакет (а может и хэш)
В std::unordered_map они разрешаются методом цепочек, каждый бакет хранит связный список элементов, про поиске обходим этот список

Другие методы:
- Linear probing все храним прям в том же массиве, при колизии ищем свободный слот
- Quadratic probing тоже самое только двигаемся на шаг в квадрате
- Double hashing берем вторую хеш функцию и двигаемся на шаг * hash2(key)

Пример коллизии

```
std::hash<int> h;
size_t h1 = h(5);      // например, 5
size_t h2 = h(1000005); // например, 1000005

size_t bucket_count = 10;
size_t bucket1 = h1 % bucket_count;  // 5
size_t bucket2 = h2 % bucket_count;  // 5
```

### 11. Что такое рехеш?

Операция пересоздания хеш таблицы с новым количеством бакетов и перераспределением всех элементов по новым позициям
Происходит автоматически при перегрузке бакетов

### 12. Есть std::vector из 100 элементов, как работает механика добавления 101 элемента?

Если capacity при этом тоже 100, то произойдет релокация
Выделяется буфер размером вдвое больше
Элементы массива перекладываются в новый буфер
Вызываются деструкторы старых элемнтов
Старый буфер освобождается
Новый элемент кладется на 101ю позицию

### 13. Когда лучше выбирать std::deque, а когда std::vector?

std::vector - непрерывный блок памяти, лучше по скорости доступа и локальности кэша
std::deque - массив блоков, эффективно вставляет и в начало и в конец, по скорости доступа и кэшу проигрывает

Итого: если нужные частые вставки в начало или стабильность ссылок на элементы - берем deque, в остальный случаех vector

### 14. Какие знаете типы итераторов? Чем они отличаются? В каких контейнерах используются?

- Input iterator только чтение, однопроходный ++, std::istream_iterator
- Output iterator только запись, однопроходный ++, std::ostream_iterator
- Forward iterator чтение + запись, многопроходный ++, std::forward_list
- Bidirectional iterator все что выше + --, std::list
- Random access iterator все что выше + +n -n [] <, std::vector

### 15. Что произойдет с итераторами std::map после перестроения дерева?

Ничего, все итераторы останутся валидными. Это происходит потому что при перестроении дерева сам элемент остается тем же, у него просто меняются указатели на родителя и детей, а красно-черное дерево гарантирует что обход будет в правильном порядке

### 16. Что произойдет с итераторами std::unordered_map после перестроении хеш-таблицы?

Все итераторы становятся невалидными. Ссылки на элементы остаются валидными, потому что узлы физически не перемещаются, но итераторы инвалидируются потому что в итераторе хранится следующий бакет и проход станет не валидным, так как бакет следующий мог поменяться

## Умные указатели

### 17. Какие умные указатели вы знаете?

- std::unique_ptr<T>
- std::shared_ptr<T>
- std::weak_ptr<T>

### 18. Расскажите про std::unique_ptr. Какими способами можно его создать? Чем эти способы отличаются?

Умный указатель с единоличным владением обьектом
Его нельзя скопировать, только переместить
При уничтожении автоматически удаляет обьект
Имеет нулевой оверхед, по размеру как сырой указатель

Способы создания:
- std::make_unique<T>(args...) предпочтительный, exception-safe
- std::unique_ptr<T> p(new T(args...)) не exception-safe
- std::unique_ptr<T, Deleter>(ptr, deleter) для ресурсов требующих очистки
- через release другого или через std::move

### 19. Зачем нужен std::unique_ptr?

Нужен для автоматического управления временем жизни обьекта в динамической памяти через RAII
Решает проблемы:
- утечка памяти, delete автоматически вызовется 
- exception safety, если бросится исключение, то деструктор все равно освободит память
- неявное владение, сразу понятно по коду кто владеет
- двойная очистка и висячий указатель, единоличное владение + move семантика

### 20. Какой размер std::unique_ptr?

По умолчанию такой же как и размер обычного указателя. Размер зависит от deleter-а, если он занимает место то итого получится размер = размер указателя + размер deleter-a

### 21. Как устроен std::unique_ptr?

Шаблонный класс-обертка над сырым указателем
Следует идиоме RAII
Владение передается только через move семантику 
Copy ctor и copy assign = delete
Для массивов есть специализация, использует delete[] и operator[]

### 22. В какой области памяти создается std::unique_ptr?

Сам обьект unique_ptr создается на стеке. Обьект, которым он владеет в куче

### 23. Можно ли использовать пользовательский удалитель в std::unique_ptr?

Да, его можно на любой callable обьект заменить, это нужно когда мало просто delete, а какая-то спец очистка
std::make_unique не поддерживает такое, надо обычный конструктор вызывать

### 24. Расскажите про std::shared_ptr. Какими способами можно его создать? Чем эти способы отличаются?

Умный указатель с совместным владением
Несколько shared_ptr могут владеть одним обьектом
Обьект уничтожается когда умирает последний указатель

Способы создания:
- std::make_shared<T>(args...) предпочтительный, exception-safe
- std::shared_ptr<T>(new T(args...)) 
- из std::unique_ptr
- копирования другого shared_ptr, увеличивает счетчик
- Aliasing constructor, то есть дать во владение только часть обьекта, а не весь
- из std::weak_ptr::lock()

### 25. В какой области памяти создается std::shared_ptr?

shared_ptr состоит их 3х сущностей:
- сам обьект обертка на стеке
- управляемый обьект в куче
- control block в куче

### 26. Какой размер std::shared_ptr?

Два указателя, то есть обычно 16 байт, один указатель на обьект, один на контрол блок, от типа deleter-а не зависит

### 27. Как устроен std::shared_ptr? Расскажите про устройство Control Block.

Обертка состоящая из двух указателей
Первый указатель на управляемый обьект
Второй указатель на контрол блок

Контрол блок - отдельная структура в куче, которая хранит:
- атомарный strong счетчик - число shared_ptr владельцев
- атомарный weak счетчик - число weak_ptr + 1, есть strong > 0
- deleter для уничтожения обьекта
- allocator для освобождения контрол блока

когда strong счетчик == 0, уничтожаем обьект
когда weak счетчик == 0, освобождаем память контрол блока

### 28. Какой тип имеют счетчики ссылок в std::shared_ptr?

Атомарные целочисленные типы, чтобы при взаимодействии из разных потоков ничего не сломалось. Но это не значит что shared_ptr thread-safe

### 29. Какой размер std::weak_ptr?

Такой как у shared_ptr - два указателя, на сам обьект и на контрол блок

### 30. Как устроен std::weak_ptr?

Хранит два указателя, на сам обьект и на контрол блок. Ключевое отличие от shared_ptr - инкрементирует weak_ptr
Не продлевает время жизни обьекта
Продлевает жизнь контрол блока
Нет прямого разыменовывания, только через lock() или expired()

### 31. Зачем нужен std::weak_ptr?

- разрыв циклических ссылок. Если цикл из shared_ptr то утечка памяти, используем один weak_ptr для разрыва ссылок
- наблюдение за жизнью обьекта

### 32. Приведите пример того, как вы в своей работе используете std::weak_ptr.

Рассмотрим пример с деревом виджетов на экране, виджет владеет всеми детьми через shared_ptr, а на своего родителя ссылается слабо. Если бы и тот были бы shared_ptr, то был бы цикл и случилась бы утечка памяти
