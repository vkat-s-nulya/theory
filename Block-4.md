# Многопоточка

## Общие вопросы

### 1. Расскажи кратко про архитектуру процессора.

Сам процессор состоит из ядер, каждое ядро выполняет какие-то инструкции
Далее в каждом ядре есть:
- регистры, они хранят операнды текущих операций
- АЛУ, выполняет сами арифметические или логические операции
- блок управления, декодирует инструкции и управляет исполнением
- иерархия кешей, для того чтобы каждой раз не ходить в RAM

### 2. Зачем процессору нужен кеш? Какие кеши бывают?

Процессору для исполнения операций нужны какие-то данные, данные хранятся в RAM. Если каждый раз за данными ходить в RAM, то ядро будет часто простаивать т.к. получение данных это медленная операция. Для этого придуман кэш, чтобы если мы запросили данные какие-то, то мы их сохранили где-то поближе чем RAM для быстрого доступа. Он основан на двух принципах: временная локальность и пространственная локальность

Кеши обычно делятся на 3 уровня и из них строится иерархия:
- L1, самый быстрый и маленький, личный для каждого ядра
- L2, средний, личный для каждого ядра
- L3, самый большой и медленный, общий для всех ядер

### 3. Расскажи, как работает виртуальная память?

Это механизм ОС, который позволяет каждому процессу видеть свое адресное пространство, изолированное от других. Реальные физические адреса скрыты, программа видит только виртуальные, а специальный механизм Memory Managemant Unit занимается их переводом в физические через таблицы страниц

### 4. Что такое страница памяти (memory pages)?

Минимальная единица, которой оперирует виртуальная память. ОС выделяет память не по байтам, а страницами памяти фиксированного размера, обычно 4 КБ. Это сделано для того чтобы таблицы страниц были меньшего размера + блоками управлять гораздо проще чем каждым байтом

### 5. Что такое memory mapped файлы?

Механизм отображения содержимого файла в виртуальное адресное пространство процесса. После такого маппинга работа с файлом превращается в работу с обычном массивом данных

### 6. Что такое cache-miss?

Ситуация, когда процессор обратился за данными в кеш, а их там не оказалось и пришлось идти на следующий уровень иерархии

### 7. Что такое page-miss? (page-fault)

Ситуация, когда при обращении к виртуальной странице она не отображена в физическую. То есть нам надо с диска в RAM загружать данные, потому что они были вытеснены

### 8.Что такое pipelining и branch prediction?

pipelining - конвейер выполнения инструкций в процессоре
Каждая инструкция проходит несколько стадий:
- загрузка из памяти
- декодирование
- исполнение
- запись результата в память
- запись результата в регистр 

Конвейер устроен так, чтобы не ждать пока вся инструкция исполнится на всех стадиях, то есть пока одна инструкция на одной стадии, на предыдущей уже сразу другая

branch prediction - когда процессор встречает условный переход, то он не знает результат сравнения, ему надо подождать пока он вычислиться. Чтобы не тупо стоять несколько тактов, процессор угадывает результат и начинает грузить инструкции по предсказанному пути
Если угадал то все круто, быстрее посчитает, если не угадал, то отбрасывает все что сделал и начинает заново с правильной ветки

### 9. Расскажи кратко про linked-time optimization.

Оптимизация программы на этапе линковки, когда компилятор видит весь код программы, а не только одну единицу трансляции. Применяется много для чего, например удаление мертвого кода, девиртуализация, инлайнинг

### 10. Что такое процесс?

С точки зрения ОС это абстракция, которая обьединяет адресное пространство, ресурсы и состояние выполнения. У процесса свой уникальный номер PID, своя вирт память, свои права доступа. Можно сказать, что процесс это запущенный экземпляр программы

### 11. Как передать информацию между несколькими процессами (IPC)?

IPC - Inter Process Communication, набор механизмов, через которые процессы общаются между собой, несмотря на то что память изолирована
Есть несколько вариантов, используются в зависимости от сценария общения, самые популярные:
- pipes
- sockets
- shared memory
- message queues
- signals

### 12. Как организовать shared memory между процессами?

Одна и та же физическая страница памяти отображается в виртуальные адресные пространства нескольких процессов. Это позволяет писать и читать в эту область памяти как в обычную. Но могут возникнуть гонки данных

### 13. Что такое изоляция ядра?

Принцип разделения адресного пространства и привелегий между ядром ОС и пользовательскими процессами. Ядро живет в защищенной области памяти, выполняется в привилегированном режиме процессора, код и данные недоступные напрямую из юзер процессов. Юзер процесс общается с ядром через строгий интерфейс системных вызовов

### 14. Что такое поток?

Единица исполнения внутри процесса. У потока есть собственный стек, состояние регистров и счетчик инструкций, но он делит с другими потоками того же процесса все остальное: адресное пространство, открытые ресурсы, сигналы и переменные окружения. Несколько потоков одного процесса могут выполняться параллельно на разных ядрах

### 15. В чем отличие потока от процесса?

Основное отличие это изоляция. Процесс изолирован от других процессов, ничего с ними не разделяет. А внутри процесса есть потоки, они как раз таки имеют общие ресурсы внутри одного процесса

### 16. В чем отличие потока от файбера?

Отличие в том кто их планирует. Потоки планируются шедулером ядра ОС, файбер шедулится самой программой без участия ядра

### 17. Что такое критическая секция?

Участок кода, который обращается к общим данным и должен выполняться только одним потоком в одно время, если несколько потоков зайдут туда параллельно, то можем получиться данные в неконсистентном состоянии

### 18. Какие проблемы многопоточного кода вы знаете?

- race condition
- data race
- deadlock
- livelock
- голодание
- борьба за лок
- слишком много потоков
- сложность отладки и повторения бага

### 19. Что такое race condition?  Как избежать состояния гонки?

Ситуация, когда состояние программы зависит от порядка выполнения потоков, то есть при разных запусках разное состояние может быть. Чтобы защититься от этого нужно сделать так чтобы операции с общими данными выполнялись атомарно или под защитой синхронизации

### 20. Что такое data race?

Ситуация, когда два или более потоков обращаются к одной ячейке памяти, минимум один из них пишет, и между обращениями нет happens-before упорядочивания

### 21. Что такое взаимная блокировка dead lock?

Ситуация, в которой два или более потоков оказываются заблокированными навсегда, потому что каждый из них ждет ресурс, удерживаемый другим. Программа не падает, а просто зависает

### 22. Что такое live lock?

Ситуация, в которой потоки не заблокированы, активно работают, но при этом не делают полезных действий. Часто возникает при попытках избежать дедлока, потоки отпускают ресурсы друг для друга и не двигаются вперед

### 23. Что такое голодание потоков?

Ситуация, когда поток не может продвинуться в работе, потому что другие потоки постоянно опережают его в борьбе за обший ресурс

### 24. Что такое атомарная операция? Как устроены атомики?

Операция, которая выполняется либо целиком либо никак. Никакой поток не может увидеть промежуточные состояния. На уровне процессора есть специальные инструкции, которые гарантируют атомарность

### 25. Что происходит, когда два ядра читают одну и ту же память?

Кеш линия загружатеся в кеш каждого ядра и оба ядра имеют копию этой линии. Никаких проблем не возникает, каждый спокойно читает свою копию

### 26. Опиши проблему false sharing? Когда она может возникнуть?

Несколько потоков пишут в разные переменные, которые физически лежат на одной кеш линии. Все работает корректно, но кеш постоянно инвалидируется и начинается постоянное обновление, что сильно замедляет производительность

### 27. Что такое lock-free алгоритм?

Алгоритм, обеспечивающий корректную работу нескольких потоков без блокирующих примитивов. Строится на атомарных операциях. Гарантируется, что хотя бы один поток обязательно завершит свою операцию за конечное число шагов, даже если остальные потоки остановились

### 28. Что такое wait-free алгоритм?

Тоже самое что и lock-free, но гарантия сильнее, гарантируется, что все потоки завершат свою операцию за конечное число шагов независимо от других потоков

### 29. Что происходит при syscall-вызове?

Программа кладет сискол и аргументы в регистры, процессор аппаратно переключается в привилегированный режим, ищется в таблице сисколов нужная функция, проверяются аргументы, затем исполняется операция и процессор возвращается в обычный режим. Из-за такого количества действий и проверок это дорогостоящая операция

### 30. Что такое NUMA?

Non Uniform Memory Access - архитектура, в которой каждый процессор имеет свою локальную память, доступ к которой быстрее чем к памяти другого процессора. Это создает неоднорость к доступу в память. Память физически разделена на NUMA узлы 

### 31. Расскажи кратко про kernel bypass и zero-copy.

kernel bypass - техника обхода ядра ОС, которая позволяет работать напрямую с железом без всяких накладных расходов от ОС

zero copy - техника передачи данных без копирования в буфферы ядра, что сильно ускоряет работу

### 32. Что такое latency?

Время от начала операции до получения ее результата. То есть сколько времени на исполнение одной операции потребовалось

## STL
### 33. Как создать поток в С++?

Есть несколько вариантов:
- std::thread, в конструктор передаем callable, до уничтожения обьекта обязательно вызвать join или detach, аргументы передаются по значению, для ссылки надо явно std::ref
- std::jthread, тоже самое но в деструкторе сам делает join

### 34. В чем отличие join от detach?

join - блокирует вызывающий поток и ждет пока запущенный поток завершится
detach - вызывающий поток продолжает работу, созданный поток продолжает работу самостоятельно в фоне и мы теряем над ним контроль, после detaсh мы больше не можем дождаться завершения

### 35. Может ли std::thread возвращать значение?

Нет, возвращаемое значение вызываемой функции просто отбрасывается, если хочется получить значение то надо реализацию писать самому

### 36. Что произойдет, если в std::thread будет сгенерировано исключение?

Если внутри потока никто его не словил, то std::terminate

### 37. Как примитивы синхронизации потоков в С++ вы знаете?

- mutex, базовый мьютекс
- shared_mutex, несколько читателей, один писатель
- lock_guard, RAII обертка над мьютексом
- unique_lock, RAII обертка, есть lock unlock
- scoped_lock, захват нескольких мьютексов сразу без дедлока
- conditional_variable, ожидание на условии
- atomic, атомарные операции
- future/promise, передача результата между потоками
- call_once, выполнить функцию ровно один раз

### 38. Что такое mutex? Зачем он нужен?

Примитив синхронизации, который гарантирует, что в защищенную секцию в каждый момент времени может войти только один поток, остальные потоки просто ждут пока поток освободит мьютекс
Нужен для защиты от race condition и атомарности сложных операций 

### 39. Как устроен mutex?

Грубо говоря, это атомарная переменная состояния свободен/захвачен + очередь ожидающих потоков

Что делает lock:
- атомарно пытается изменить состояние свободен на захвчаен
- если справился, то захватили и прододжаем работу
- если не получилось, то кладем в очередь потоков и усыпляем поток

Что делает unlock:
- атомарно ставит состояние в свободен
- если есть ждущие, то будит одного из них
- разбуженный поток пытается захватить лок

### 40. Расскажи подробнее, как устроен spin-lock?

Примитив синхронизации, который при попытке захвата захваченного лока не усыпляет поток, а крутится в активном цикле, постоянно проверяя состояние через атомик операции

### 41. Что такое std::async?

Функция запуска задачи асинхронно. Принимает callable, возвращает std::future - обьект через который можно получить результат вычисления. При исключениях не падает, а кладет их в std::future

### 42. Какие политики запуска std::async вы знаете?

1. std::launch::async гарантировано создать новый поток и сразу начать выполнение задачи
2. std::launch::deferred не создавать поток вообще, выполнить задачу лениво в момент вызова get() или wait() в том же потоке, который вызвал эти функции
3. std::launch::async | std::launch::deferred на усмотрение рантайма, дефолтная, не рекомендуется

### 43. Как std::async реализован под капотом?

1) создается shared state
это структура в куче, через которую взаимодействуют вызывающий и исполняющий поток
Внутри в ней лежат:
- результат исполнения
- std::exception_ptr если задача бросила исключение
- флаг готов/не готов
- мьютекс и conditinal_variable для ожидания
- счетчик ссылок
2) создается packaged_task
callable заворачивается в структуру, которая связана с shared state. При вызове задача выполнит callable и запишет результат в shared state, затем нотифаит conditional_variable
3) возвращается std::future, внутри которого хранится указатель на shared state

### 44. Как работать с std::conditional_variable?

Это примитив синхронизации, который позволяет одному потоку ждать события, а другому его уведомлять, работает только в паре с мьютексом
делаем cv.wait(lock, predicate), когда предикат станет верен, то надо сделать cv.notify_one() чтобы разбудить один поток, или cv.notify_all() чтобы разбудить все потоки

### 45. Какие проблемы могут возникнуть при работе с std::conditional_variable?

- поток может проснуться без нотификации
- notify случился до захода в wait
- если мьютекс не передать, то может быть дата рейс, ждущий может пропустить нотифай
- если сделать notify_one когда нужен notify_all, то проснется только один а остальные зависнут
- если сделать notify_all когда нужен notify_one, то проснуться все, начнут толькаться за мьютекс, лишние контекст свичи

### 46. Что такое CAS (для атомиков)?

Compare And Swap - атомарная операция, которая меняет значение, только если оно равно предполагаемому, если так и есть то возвращает true, иначе false

### 47. Расскажите про потокобезопасность std::shared_ptr.

Счетчик ссылок атомарный, потокобезопасный
Сам обьект не потокобезопасный, ничем не защищен

### 48. Расскажи про модель памяти C++ (имеется в виду std::memory_order)

Модель памяти описывает что один поток видит из того, что записал другой
std::memory_order - параметр операций, который задает какие гарантии нужны

relaxed только атомарность, никаких ограничений на порядок
acquire/release для публикации данных, все что записано до release видно поcле acquire
acq_rel для read-modify-write
seq_cst - дефолт, единый глобальный порядок для seq_cst операций, все потоки видят один и тот же порядок
